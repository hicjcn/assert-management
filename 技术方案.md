# 资产管家技术方案

## 一、技术栈定稿

本项目定位为手机优先的自部署 Web App，主要使用 iPhone Safari 访问，也支持添加到主屏幕作为类 App 使用。

最终技术栈：

- 前端与服务端框架：Next.js + TypeScript
- UI：Tailwind CSS + shadcn/ui
- 图表：ECharts
- 表单与校验：React Hook Form + Zod
- 数据库 ORM：Prisma
- MySQL 连接：Prisma Driver Adapter，使用 `@prisma/adapter-mariadb` 和 `mariadb`
- 数据库：MySQL，使用外部自建数据库或云数据库
- 登录认证：NextAuth/Auth.js 或自实现 Cookie Session
- 部署：Dockerfile 构建单应用镜像
- 反向代理：Nginx、Caddy 或云厂商负载均衡
- 运行环境：Node.js LTS

明确不使用：

- Docker Compose
- 本地浏览器 IndexedDB 作为主数据源
- 原生 iOS App 上架
- React Native / Expo
- 桌面端 Tauri

## 二、方案目标

技术方案需要满足：

- iPhone 上操作顺手，页面以移动端为第一优先级。
- 数据存储在自己的 MySQL 中，不依赖浏览器本地存储。
- 能部署到自己的云服务器。
- 应用服务只需要一个 Docker 镜像。
- 数据库可以是自建 MySQL，也可以是云数据库。
- 第一版保持简单，后续可以扩展云备份、PWA、访问控制和多用户。

## 三、整体架构

```text
iPhone Safari / 主屏幕快捷方式
        |
      HTTPS
        |
Nginx / Caddy / 云负载均衡
        |
Next.js Docker 容器
        |
Prisma
        |
外部 MySQL / 云数据库
```

说明：

- 用户通过 HTTPS 访问 Web App。
- Next.js 同时负责页面渲染、接口、服务端逻辑。
- Prisma 负责数据库访问和迁移。
- MySQL 不随应用容器启动，由用户单独部署或购买云数据库。
- 应用容器通过环境变量读取数据库连接信息。

## 四、应用形态

### 4.1 手机优先 Web App

第一版按移动端页面设计：

- 页面宽度优先适配 iPhone。
- 桌面端可以正常访问，但不是第一优先级。
- 主要操作入口放到底部导航或顶部快捷按钮。
- 表单输入适配手机键盘。
- 图表在小屏幕上优先保证可读性。

### 4.2 PWA 支持

第一版可以支持基础 PWA：

- 添加 manifest。
- 配置应用名称、图标、主题色。
- 支持 Safari 添加到主屏幕。

注意：

- PWA 只作为入口体验增强。
- 核心数据仍然存在 MySQL。
- 不依赖浏览器本地缓存保存资产数据。

核心模块：

- 账户模块：账户新增、编辑、归档、列表。

## 七、数据库设计

### 7.1 用户表

第一版可以只支持一个用户，但仍建议保留用户表，方便后续扩展。

字段建议：

- id
- username
- password_hash
- created_at
- updated_at

### 7.2 账户表 accounts

字段建议：

- id
- user_id
- name
- category
- account_type，`asset` 或 `liability`
- current_amount
- include_in_stats
- archived
- note
- created_at
- updated_at

说明：

- `current_amount` 使用整数分存储，例如 123.45 元存为 12345。
- 负债账户页面展示为欠款金额，统计时按负债处理。
- `include_in_stats = false` 的账户不参与首页和图表统计。

### 7.3 变更记录表 account_changes

字段建议：

- id
- user_id
- account_id
- account_name_snapshot
- category_snapshot
- change_type
- before_amount
- change_amount
- after_amount
- note
- changed_at
- created_at

说明：

- 金额字段统一使用整数分。
- 账户名称和分类保存快照，避免账户改名后历史记录失真。
- 每次账户金额变化都必须写入一条记录。

### 7.4 目标表 goals

字段建议：

- id
- user_id
- name
- target_amount
- one_time_income
- one_time_expense
- note
- created_at
- updated_at

说明：

- 目标表只保存每个目标自身的金额与备注。
- 目标当前已完成金额不落库，由关联账户余额实时计算；没有关联账户时使用总净资产。
- 每月收入、房租、餐饮、日常支出、其他收支通常是固定配置，统一保存到 `settings`，避免每个目标重复维护。

### 7.5 目标账户关联表 goal_accounts

字段建议：

- goal_id
- account_id

说明：

- 一个目标可以关联一个或多个未归档资产账户。
- 没有关联记录表示该目标使用总净资产计算进度。
- 关联账户归档后不再计入当前已完成金额，但保留关联关系，提示用户重新选择。

### 7.6 系统设置表 settings

字段建议：

- id
- user_id
- key
- value
- created_at
- updated_at

可用于保存：

- 默认图表时间范围
- 是否启用 PWA 提示
- 金额展示偏好
- 上次导出时间
- 目标计划共用的月度收支配置

## 八、Prisma 模型方向

数据库 schema 使用 Prisma 管理。

设计原则：

- 金额统一用 `BigInt` 或 `Int` 存储分，不用浮点数。
- 时间统一用数据库时间字段。
- 表之间保留用户隔离字段 `user_id`。
- 删除账户优先用归档，不做物理删除。
- 变更记录不可随意删除，错误通过修正记录解决。

迁移方式：

- 开发环境使用 `prisma migrate dev`。
- 生产环境使用 `prisma migrate deploy`。
- 容器启动时不自动强制迁移，避免误操作生产数据库。

## 九、认证与安全

第一版建议做单用户登录。

基础要求：

- 未登录不能访问任何业务页面和接口。
- 密码使用 bcrypt 或 argon2 哈希保存。
- 登录状态使用 httpOnly Cookie。
- Cookie 设置 `Secure`、`SameSite=Lax`。
- 登录失败需要做简单频率限制。
- 提供修改密码功能。

可选增强：

- 只允许指定用户名登录。
- 后台配置初始化管理员。
- 支持 Cloudflare Access、Tailscale 或服务器 IP 白名单。
- 管理后台关闭公开注册。

部署安全要求：

- 必须启用 HTTPS。
- MySQL 不开放公网，或者只允许应用服务器 IP 访问。
- 数据库账号只授予当前库权限。
- `.env` 不提交到 Git。
- 备份文件不放在 Web 可访问目录。

## 十、API 设计

可以优先使用 Next.js Route Handler 或 Server Actions。

建议原则：

- 表单提交类操作走 Server Actions。
- 查询列表和图表数据可以用 Route Handler。
- 所有写操作在服务端校验权限和参数。
- 金额计算全部在服务端复核。

主要接口：

| 功能 | 方法 | 路径 |
| --- | --- | --- |
| 登录 | POST | `/api/auth/login` |
| 退出 | POST | `/api/auth/logout` |
| 账户列表 | GET | `/api/accounts` |
| 新增账户 | POST | `/api/accounts` |
| 更新账户 | PATCH | `/api/accounts/[id]` |
| 归档账户 | POST | `/api/accounts/[id]/archive` |
| 新增变更 | POST | `/api/account-changes` |
| 变更记录 | GET | `/api/account-changes` |
| 首页汇总 | GET | `/api/dashboard` |
| 图表数据 | GET | `/api/charts` |
| 目标列表 | GET | `/api/goals` |
| 新增目标 | POST | `/api/goals` |
| 更新目标 | PATCH | `/api/goals/[id]` |
| 导出数据 | GET | `/api/export` |

## 十一、统计计算

统计计算优先在服务端完成。

首页指标：

- 总资产：统计正资产账户金额。
- 总负债：统计负债账户金额绝对值。
- 净资产：总资产减总负债。
- 本月变化：当前净资产减上月月末净资产。
- 目标进度：当前完成金额除以目标金额。
- 计划达成：根据统一配置的月度收支净额估算。
- 趋势达成：按目标关联账户最近 6 个自然月的实际净增长月均值估算；未关联账户时使用总净资产口径。
- 趋势统计将账户初始金额计入创建当月；负债账户的初始金额及后续变化按净资产方向反向计算。

走势图：

- 默认最近 12 个月。
- 每个月取月末金额。
- 如果当月没有变更，沿用最近一次金额。
- 负债图表展示绝对值。

排行榜：

- 资产账户按金额降序。
- 负债账户按欠款金额降序。
- 净资产贡献按账户净贡献降序。

## 十二、部署方案

### 12.1 Dockerfile 单镜像

应用只提供 Dockerfile。

部署流程：

```text
拉取代码
构建 Docker 镜像
配置环境变量
运行容器
反向代理转发 HTTPS 流量到容器
```

容器只包含：

- Next.js 应用
- Prisma Client
- 静态资源
- 运行时 Node.js

容器不包含：

- MySQL
- 备份服务
- 反向代理

### 12.2 环境变量

必需环境变量：

```text
DATABASE_URL=mysql://user:password@host:3306/assert_management
AUTH_SECRET=replace-with-long-random-secret
APP_URL=https://your-domain.example.com
```

可选环境变量：

```text
INITIAL_ADMIN_USERNAME=admin
INITIAL_ADMIN_PASSWORD=change-me-on-first-login
EXPORT_MAX_ROWS=100000
```

### 12.3 推荐运行命令

示例：

```bash
docker build -t asset-management:latest .
docker run -d \
  --name asset-management \
  --restart unless-stopped \
  -p 3000:3000 \
  --env-file .env.production \
  asset-management:latest
```

### 12.4 反向代理

可以使用 Nginx、Caddy 或云厂商负载均衡。

要求：

- 对外只暴露 HTTPS。
- 反向代理到容器的 `3000` 端口。
- 配置合理的请求体大小限制。
- 可选开启基础访问日志。

## 十三、数据库部署要求

如果使用自建 MySQL：

- MySQL 与应用服务尽量放在同一区域。
- 只允许应用服务器访问数据库端口。
- 开启定期备份。
- 生产库账号只给当前库权限。

如果使用云数据库：

- 开启白名单，只放通应用服务器 IP。
- 开启自动备份。
- 开启慢查询日志。
- 选择 MySQL 8 或兼容版本。

## 十四、备份与恢复

资产数据非常重要，第一版必须考虑备份。

基础备份：

- 数据库每日自动备份。
- 应用内支持 CSV 导出。
- 设置页展示最近一次导出时间。

建议备份内容：

- accounts
- account_changes
- goals
- settings

恢复策略：

- 第一版可以先支持手工导入数据库备份。
- 后续再做应用内 CSV/Excel 导入。

## 十五、开发顺序

建议开发顺序：

1. 初始化 Next.js、TypeScript、Tailwind、基础布局。
2. 接入 Prisma 和 MySQL。
3. 完成登录认证。
4. 完成账户表、变更记录表、目标表。
5. 实现账户列表和新增账户。
6. 实现金额变更并自动写入记录。
7. 实现首页汇总。
8. 实现统计图表。
9. 实现目标计划。
10. 实现数据导出。
11. 补齐 Dockerfile 和部署文档。

## 十六、第一版验收标准

第一版完成时需要满足：

- iPhone Safari 可正常访问和操作。
- 登录后才能看到资产数据。
- 可以新增账户、编辑账户、归档账户。
- 可以对账户做增加、减少、直接设定。
- 每次金额变化都有变更记录。
- 首页能展示总资产、总负债、净资产、目标进度。
- 图表能展示趋势、占比、排行榜。
- 目标计划能计算预计达成时间。
- 数据存在 MySQL 中。
- 应用可以通过 Dockerfile 构建镜像并运行。
- 支持至少 CSV 导出备份。
